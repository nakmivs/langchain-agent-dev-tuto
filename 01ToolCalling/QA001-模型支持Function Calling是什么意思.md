# QA001：模型支持 Function Calling 是什么意思？不支持是没训练过吗？

## 问题

有的模型写的支持 Function Calling，有的不支持，是什么意思？模型只要能明白工具的结构，并 JSON 输出调用参数就可以了吧？不支持是不是说模型没训练过这种结构理解？

## 回答

"支持"与"不支持"的差异体现在三个层次：

### 层次 1：API 协议层

支持的模型，API 提供了专门的字段：

- 请求时有 `tools` 参数传入工具定义
- 响应时有 `tool_calls` 字段返回结构化的调用请求（不是纯文本）

不支持的模型，API 没有这些字段，只能返回纯文本。

### 层次 2：训练层（核心差异）

支持 Function Calling 的模型经过了**专门的微调训练**，在训练数据中见过大量工具调用模式。

核心区别不是"能不能输出 JSON"，而是**三个关键判断力**：

| 能力 | 训练过 | 未训练 |
|------|--------|--------|
| **何时调用** | 精准判断什么时候该用工具、什么时候直接回答 | 该调不调、不该调乱调 |
| **选哪个** | 在多个工具中准确选择最合适的 | 容易选错 |
| **参数生成** | 稳定生成符合 Schema 的参数 | 类型/格式不稳定，可能编造参数 |

### 层次 3：解码层（结构化输出）

结构化输出通过**约束解码（Constrained Decoding）**解决格式问题——在生成每个 token 时，只允许从"能构成合法 JSON"的 token 中选择，从机制上保证输出一定是合法 JSON。

### 结论

- 不支持的模型也能通过 prompt 模拟 Tool Calling，但可靠性远不如原生支持
- 类比：学过开车的人 vs 看过开车视频的人——都"知道"怎么开，但可靠性天差地别
- 结构化输出是更通用的能力（保证合法 JSON），Function Calling 是它的一种具体应用
