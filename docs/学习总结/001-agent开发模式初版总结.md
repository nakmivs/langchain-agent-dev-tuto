# Agent 开发模式初版总结

## 一个核心判断

你现在的理解是对的，而且很接近 agent 开发的本质：

**很多所谓的 agent 技巧，最后都会收敛到一件事：在合适的时机，给模型最相关、最精确、最必要的上下文。**

如果上下文错了、杂了、晚了，模型能力再强也容易跑偏。  
如果上下文选得准，很多复杂技巧其实都可以简化。

但再往前走一步，可以把这个理解扩成一句更完整的话：

**Agent 开发 = 状态驱动下的上下文管理。**

也就是说：

- `状态` 决定当前系统处于哪一步、已经做过什么、接下来该做什么
- `上下文` 决定这一轮该把什么信息交给模型

上下文是给模型看的，状态是给系统自己保存和推进流程用的。

## 为什么“上下文管理”是核心

LLM 本身并不会长期、稳定、结构化地记住系统过程。它每一轮都更像是：

- 读取你这次给它的输入
- 基于输入做推断
- 产出一个结果

所以 agent 是否可靠，往往不取决于你 prompt 写得多花，而取决于：

1. 你是否知道当前任务阶段是什么
2. 你是否知道模型此刻真正需要哪些信息
3. 你是否能把无关信息挡在上下文之外
4. 你是否能把中间结果稳定保存，而不是全靠模型“记住”

这也是为什么很多工程经验最后都会指向：

- RAG 是在补上下文
- Memory 是在筛选和组织上下文
- Tool calling 是在按需获取上下文
- Planning/ReAct 是在控制上下文的生成时机
- Workflow/Graph 是在控制上下文切换的顺序

## 状态和上下文要分开想

这是很重要的一层抽象。

### 1. 状态不是上下文本身

状态可以包含很多东西：

- 当前任务目标
- 已完成的步骤
- 工具调用结果
- 中间产物
- 错误信息
- 用户偏好
- 外部资源定位信息

但这些内容不一定都要原样塞进 prompt。

一个常见错误是：

**把全部状态直接拼进上下文。**

这样会导致：

- token 浪费
- 重点淹没
- 模型注意力分散
- 老信息污染当前决策

更合理的方式是：

**状态是总仓库，上下文是按当前节点裁剪后的视图。**

### 2. 状态可以存在文件系统

完全可以，而且这是很实用的一种方式。

状态不一定非要在内存里，也不一定非要放数据库。它可以存在：

- 进程内内存
- Redis
- SQLite / Postgres
- 向量库
- 对象存储
- 文件系统

文件系统特别适合这些场景：

- 单机开发和调试
- 需要保留中间产物
- 希望 agent 能“落地”生成文档、计划、草稿、缓存
- 需要可审计、可回放、可人工介入

比如可以把状态拆成：

- `task.json`：任务元信息
- `messages.jsonl`：关键对话和决策日志
- `artifacts/`：中间产物
- `scratchpad.md`：阶段性思考
- `checkpoints/`：流程检查点

这样做的价值不是“把文件当数据库”，而是：

- 它天然可见
- 易于调试
- 易于恢复
- 易于人工修改
- 易于和工作流结合

## 我目前会把常见模式归成 4 类

## 1. 状态管理模式

核心问题：**系统现在走到哪了？**

典型做法：

- 用结构化 state 表示当前任务
- 给每一步定义输入、输出、失败处理
- 保存关键中间结果，而不是只保存最终答案
- 支持 checkpoint 和 resume

关键词：

- workflow state
- graph state
- durable execution
- checkpoint

## 2. 上下文组装模式

核心问题：**这一轮到底该给模型看什么？**

典型做法：

- 区分系统规则、用户目标、任务状态、外部知识、历史记录
- 按节点动态拼装 prompt
- 只注入当前步骤真正需要的信息
- 让工具结果先结构化，再决定哪些要暴露给模型

关键词：

- prompt assembly
- retrieval
- memory selection
- context compression

## 3. 工具使用模式

核心问题：**模型缺的信息，从哪里补？**

典型做法：

- 把工具当成“获取上下文”的能力，而不是魔法
- 能先检索就不要先生成
- 能先观察就不要先行动
- 工具返回结果尽量结构化，便于后续裁剪

关键词：

- tool calling
- observation
- structured output
- action-result loop

## 4. 流程控制模式

核心问题：**什么时候该思考，什么时候该检索，什么时候该结束？**

典型做法：

- 把开放式 agent 收敛成有限状态流转
- 给不同节点设不同职责
- 为终止条件、重试条件、人工介入条件设边界
- 用 graph/workflow 管理分支，而不是全压给单轮 prompt

关键词：

- router
- planner
- executor
- reviewer
- human-in-the-loop

## 一个更实用的判断框架

以后看到一个 agent 技术点，可以先问 4 个问题：

1. 它是在管理状态，还是在管理上下文？
2. 它是在生产信息，还是在筛选信息？
3. 它是在帮助决策，还是在帮助执行？
4. 它有没有降低系统的不确定性？

如果一个技巧不能让系统在这些问题上更清晰，它大概率只是“看起来高级”。

## 一个很值得长期坚持的工程观

不要把 agent 看成“一个会自己思考的大 prompt”，更应该把它看成：

**一个由状态机、上下文构造器、工具接口、模型节点共同组成的系统。**

其中：

- 状态机负责推进流程
- 上下文构造器负责喂给模型正确信息
- 工具负责连接外部世界
- 模型负责在不确定场景下做语义判断

这四者里，模型当然重要，但真正决定系统稳定性的，往往是前三者。

## 当前这版总结的结论

可以先把你的理解收束成下面这句话：

**Agent 开发最核心的不是“让模型更聪明”，而是让系统在正确阶段，以正确形式，把正确信息交给模型。**

进一步说：

- 状态决定流程
- 上下文决定质量
- 工具决定信息来源
- 工作流决定这些东西如何协同

如果后面继续总结，我建议第二版可以专门展开：

1. `状态` 和 `记忆` 的区别
2. `工作流` 和 `自主 agent` 的边界
3. LangChain / LangGraph 里这些抽象分别落在哪
