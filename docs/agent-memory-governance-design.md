# 智能体记忆治理设计

## 一、背景

在实际落地中，智能体既需要稳定的规则约束，也需要随着使用不断吸收经验。

问题在于，这两类知识的来源并不相同：

- 有些规则在系统上线前就已经明确，适合直接写入系统提示词、配置文件或固定文档。
- 有些规则在一开始并不明确，需要内部人员通过试用、调教、复盘逐步沉淀。
- 外部用户在使用过程中也会提供大量输入，但出于安全考虑，不应默认允许他们改变智能体的长期行为。

因此，需要一个分层的记忆治理方案：既允许内部持续优化智能体，又防止外部用户污染长期记忆。

---

## 二、核心目标

这个方案希望同时满足以下目标：

1. **行为稳定**：核心安全规则、输出风格和底线不因日常对话漂移。
2. **支持演进**：允许内部人员逐步调教智能体，沉淀经验。
3. **权限隔离**：外部用户可以使用已有经验，但不能直接写入长期记忆。
4. **可审计**：长期记忆的形成过程应可回溯，避免“聊着聊着就被永久写坏”。
5. **易于实现**：先用简单机制上线，后续再逐步演进。

---

## 三、设计原则

### 3.1 固定规则和可演化经验分开

以下内容应优先放在固定规则层，而不是长期记忆中：

- 安全边界
- 禁止行为
- 必须遵守的输出规范
- 稳定的异常处理总原则

长期记忆更适合承载：

- 逐步总结出来的输出偏好
- 高频异常场景的经验性处理方式
- 业务术语解释
- 模板、示例、最佳实践

### 3.2 长期记忆的核心是“持久化”，不是“必须来自对话”

长期记忆的关键特征是：**能跨会话保存并再次读取**。

因此，一个知识项既可以：

- 来自内部人员与智能体的交互沉淀
- 也可以由团队提前整理后直接写入记忆存储

区别不在于“它是不是后来聊出来的”，而在于“它是否被纳入长期可复用的记忆层”。

### 3.3 写权限必须受控

不是所有用户都应该有权影响长期记忆。

推荐的原则是：

- 内部人员：允许提出和维护长期记忆
- 外部人员：默认只读，不允许写入
- 系统自动写入：默认关闭，至少不能无审核全自动写入

## 四、分层架构

### 4.1 四层知识结构

可以将智能体使用的知识分成四层：

| 层级 | 作用 | 来源 | 是否可跨会话 | 是否允许外部写入 |
|---|---|---|---|---|
| **固定规则层** | 定义底线、风格、硬约束 | system prompt / AGENTS.md / policy / skills | 是 | 否 |
| **长期记忆层** | 沉淀经验、偏好、模板、业务知识 | 内部维护、审核入库 | 是 | 否 |
| **会话上下文层** | 服务当前任务，保留短期状态 | 当前对话、工具结果 | 否（默认） | 是，但仅限当前会话 |
| **工具/知识库层** | 提供可检索外部资料和结构化知识 | 文档库、数据库、向量库 | 是 | 按系统权限控制 |

### 4.2 各层职责边界

#### 固定规则层

这一层解决“必须稳定”的问题。适合放：

- 回答风格约束
- 安全要求
- 不能违反的业务规则
- 异常处理的大原则

特点是：

- 上线前已知
- 变化频率低
- 必须可审计
- 不依赖用户互动自动形成

#### 长期记忆层

这一层解决“经验沉淀”的问题。适合放：

- 某类任务更合适的输出模板
- 某类异常更稳妥的兜底方式
- 业务内部约定术语
- 内部试用后总结出的高频偏好

特点是：

- 可以逐步累积
- 需要筛选和治理
- 可跨会话复用
- 应与固定规则分开管理

#### 会话上下文层

这一层只服务当前会话，例如：

- 当前用户的问题
- 当前任务的中间结果
- 本轮工具调用输出
- 临时澄清条件

特点是：

- 生命周期短
- 可被总结和压缩
- 默认不应提升为长期记忆

## 五、权限模型

### 5.1 推荐角色划分

| 角色 | 读取固定规则 | 读取长期记忆 | 写长期记忆 | 影响当前会话 |
|---|---|---|---|---|
| **内部调教人员** | 是 | 是 | 是（建议经审核） | 是 |
| **普通内部用户** | 是 | 是 | 视权限而定 | 是 |
| **外部用户** | 是（间接生效） | 是（间接生效） | 否 | 是 |

### 5.2 外部用户的约束

外部用户与智能体的对话可以：

- 影响当前回答
- 影响当前线程内的临时上下文
- 触发常规工具调用

但不应默认允许：

- 把偏好永久写入长期记忆
- 修改底层规则
- 通过提示注入改变未来所有用户的默认体验

这就是“外部可使用，外部不可训练”的核心。

## 六、长期记忆写入流程

### 6.1 不建议“聊过就自动永久记住”

如果内部人员说了一句话，系统就直接把它写成长期记忆，会有几个风险：

- 偶然表述被永久固化
- 相互矛盾的记忆不断累积
- 错误经验进入系统后难以发现
- 同一规则在不同地方重复出现

### 6.2 推荐流程：提议 -> 审核 -> 入库

更稳妥的方式是：

1. 内部人员通过对话或标注提出“候选记忆”。
2. 系统将其整理成结构化条目。
3. 由人工或审核工作流确认是否采纳。
4. 审核通过后写入长期记忆层。
5. 后续回答时，智能体按需检索并使用这些记忆。

### 6.3 候选记忆示例

```json
{
  "category": "output_style",
  "title": "异常处理回答格式",
  "content": "当接口调用失败且无法定位具体原因时，优先按“现象、可能原因、排查步骤、建议动作”四段式输出。",
  "source": "internal_training",
  "status": "proposed",
  "reviewer": null
}
```

这个结构比“直接塞进 prompt”更便于治理。

## 七、一个可行的产品形态

### 7.1 面向内部

提供一个“调教模式”或“内部训练模式”：

- 可查看当前固定规则和长期记忆摘要
- 可提出新的规则或经验
- 可对已有记忆进行修订、合并、废弃
- 可通过评估集验证新记忆是否真的改善效果

### 7.2 面向外部

提供一个“使用模式”：

- 默认加载固定规则和已批准的长期记忆
- 只保留当前会话所需的临时上下文
- 不开放长期记忆写入权限
- 必要时可开启更严格的安全过滤和审计日志

## 八、与 Deep Agents / LangGraph 的对应关系

### 8.1 固定规则层

可放在以下位置：

- `system_prompt`
- `AGENTS.md`
- skills
- 固定 policy 文件

### 8.2 长期记忆层

可放在以下位置：

- LangGraph Store
- `CompositeBackend` 映射出的长期存储路径
- 独立数据库或知识库存储

核心要求不是存哪里，而是：

- 可持久化
- 可检索
- 可审计
- 可区分读写权限

### 8.3 会话上下文层

由运行中的 agent 自动维护，例如：

- 当前线程状态
- 工具调用结果
- 总结后的历史上下文

这部分应允许压缩、总结和过期，而不应默认进入长期记忆。

---

## 九、推荐的最小可行实现

如果现在就要做一个可用版本，建议按以下步骤落地：

1. 把稳定规则先写入固定提示词或 `AGENTS.md`。
2. 新建一个“内部记忆库”，只允许内部人员维护。
3. 外部用户会话只读取该记忆库，不允许写入。
4. 对内部新增记忆采用“候选 -> 审核 -> 生效”的流程。
5. 先记录少量高价值记忆，例如输出格式、常见异常处理、业务术语表。

这样可以先把治理框架搭起来，再决定是否进一步做：

- 自动候选记忆抽取
- 记忆去重和冲突检测
- 记忆版本管理
- 基于评测集的回归验证

---

## 十、最终结论

这个想法是可行的，而且很适合作为企业或团队内部智能体的标准方案。

它的本质不是“让智能体自由长记忆”，而是：

**把智能体的知识分层，并把长期记忆的写权限纳入治理。**

最关键的设计结论有三点：

1. **固定规则优先放在提示词和规则文件中，不依赖长期记忆兜底。**
2. **长期记忆用于沉淀经验，但写入必须受控，最好经过审核。**
3. **外部用户可以消费长期记忆带来的能力提升，但不应默认拥有长期记忆写权限。**

这能在“持续优化”和“安全稳定”之间取得比较好的平衡。
